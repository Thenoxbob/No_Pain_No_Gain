Bases des Réseaux

Les fondations indispensables : comprendre comment les machines communiquent, s'identifient et échangent des données.

01 Qu'est-ce qu'un réseau informatique ?

Un réseau informatique est un ensemble de machines (ordinateurs, serveurs, smartphones, imprimantes…) interconnectées dans le but de partager des ressources et d'échanger des données. Ces connexions peuvent être filaires (câbles Ethernet, fibre) ou sans fil (Wi-Fi, Bluetooth, 4G/5G).

Un réseau repose sur trois composantes fondamentales : les nœuds (les équipements), les liens (les connexions entre eux), et les protocoles (les règles de communication communes).

Le principe de base est simple : chaque machine parle un même langage standardisé pour se comprendre, peu importe son fabricant ou son système d'exploitation. Un iPhone peut donc accéder à un serveur Linux hébergé sous Windows Azure — grâce aux protocoles.

// concept clé
Un protocole est un ensemble de règles que les machines respectent pour se comprendre. Sans protocoles standardisés (TCP/IP, HTTP, DNS…), un Mac ne pourrait pas communiquer avec un PC Windows ou un serveur Linux. 
# Commandes de diagnostic réseau de base

# Tester la connectivité (couche 3 — réseau)
$ ping 192.168.1.1
$ ping google.com

# Tracer la route d'un paquet jusqu'à destination
$ traceroute google.com       # Linux / macOS
$ tracert google.com          # Windows

# Afficher les interfaces réseau et leurs adresses IP
$ ip addr show                # Linux
$ ipconfig /all               # Windows
$ ifconfig                    # macOS / Linux (ancien)

# Afficher la table de routage
$ ip route show
$ netstat -rn

03 Topologies réseau

La topologie décrit comment les équipements sont physiquement ou logiquement organisés et connectés. Le choix de la topologie a des conséquences directes sur la résilience, les performances, les coûts et la facilité de dépannage.

// architecture d'entreprise standard
Le modèle 3-tiers de Cisco est la référence : Core (cœur de réseau haute vitesse), Distribution (agrégation, routage inter-VLAN, politiques de sécurité), Access (switches connectés aux postes de travail). Cette hiérarchie simplifie l'administration et isole les pannes.

04 Le Modèle OSI

Le modèle OSI (Open Systems Interconnection), publié par l'ISO en 1984, découpe les communications réseau en 7 couches abstraites. Chaque couche a un rôle précis et ne communique qu'avec les couches directement adjacentes.

Comprendre ce modèle est essentiel pour diagnostiquer les pannes : si le ping fonctionne (couche 3) mais pas HTTPS (couche 7), le problème est applicatif — pas réseau.

6PrésentationFormatage des données, chiffrement, compressionSSL/TLS, JPEG, MPEG, ASCII, UnicodeMessage
5SessionÉtablissement, maintien et fermeture des sessionsNetBIOS, RPC, PPTP, SOCKSMessage
4TransportSegmentation, contrôle de flux, fiabilité bout-à-boutTCP, UDP, SCTPSegment / Datagramme

// mémos
De haut en bas (7→1) : "Ah, Peu De Techniciens Réparent Les Pannes"
De bas en haut (1→7) en anglais : "Please Do Not Throw Sausage Pizza Away"
// OSI vs TCP/IP
En pratique, on parle du modèle TCP/IP à 4 couches : Application (OSI 5-6-7), Transport (OSI 4), Internet (OSI 3), Accès réseau (OSI 1-2). Le modèle OSI reste la référence théorique pour l'enseignement et le diagnostic des pannes.

05 TCP/IP & protocoles essentiels

TCP (Transmission Control Protocol) garantit la livraison fiable et ordonnée des données via un processus de connexion en 3 étapes. Il est utilisé pour le web (HTTP/S), les emails, les transferts de fichiers (FTP/SFTP), SSH.

UDP (User Datagram Protocol) est sans connexion et sans garantie de livraison. Il privilégie la vitesse sur la fiabilité — idéal pour le streaming vidéo, les jeux en ligne (où un paquet perdu est préférable à un paquet en retard), la VoIP, le DNS.

# Le Three-Way Handshake TCP (établissement de connexion)
Client  → Serveur : SYN      # "Je veux me connecter, mon seq=1000"
Serveur → Client  : SYN-ACK  # "OK, mon seq=5000, j'attends seq=1001"
Client  → Serveur : ACK      # "Reçu, j'attends seq=5001 — connexion ouverte !"

# Fermeture propre TCP (4-way handshake)
Client  → Serveur : FIN      # "Je n'ai plus rien à envoyer"
Serveur → Client  : ACK
Serveur → Client  : FIN      # "Moi non plus"
Client  → Serveur : ACK      # Connexion fermée

06 Adressage IPv4

Une adresse IPv4 est un identifiant numérique de 32 bits, noté en quatre octets séparés par des points. Exemple : 192.168.1.42 (chaque nombre est compris entre 0 et 255).

Chaque adresse se divise en deux parties : la partie réseau (identifie le réseau) et la partie hôte (identifie la machine). Le masque de sous-réseau détermine où se trouve la frontière.

L'IANA a alloué ses derniers blocs IPv4 en 2011. Les adresses publiques sont épuisées. C'est pourquoi le NAT est omniprésent (plusieurs machines partagent une seule IP publique) et la migration vers IPv6 s'accélère chaque année.

07 IPv6

IPv6 est la version suivante du protocole IP, conçue pour pallier l'épuisement des adresses IPv4. Une adresse IPv6 est codée sur 128 bits, notée en 8 groupes de 4 chiffres hexadécimaux séparés par des deux-points.

Exemple complet : 2001:0db8:85a3:0000:0000:8a2e:0370:7334

Forme abrégée (groupes de zéros consécutifs remplacés par ::) : 2001:db8:85a3::8a2e:370:7334

08 Subnetting (Sous-réseaux)

Le subnetting consiste à découper un espace d'adressage IP en plusieurs sous-réseaux plus petits. C'est une compétence fondamentale : tous les examens réseau (CCNA, Network+) y consacrent une partie importante.

La notation CIDR (Classless Inter-Domain Routing) indique combien de bits servent à identifier le réseau. Un /24 signifie 24 bits réseau → 8 bits pour les hôtes → 2⁸ − 2 = 254 hôtes utilisables.

# Exemple : découper 192.168.10.0/24 en 4 sous-réseaux égaux → /26

Adresse de départ  : 192.168.10.0/24
Objectif           : 4 sous-réseaux
Nouveau masque     : /26  (255.255.255.192)
Hôtes par sous-réseau : 62  (2^6 - 2)

Sous-réseau 1 : 192.168.10.0/26    hôtes .1 → .62    broadcast .63
Sous-réseau 2 : 192.168.10.64/26   hôtes .65 → .126  broadcast .127
Sous-réseau 3 : 192.168.10.128/26  hôtes .129 → .190 broadcast .191
Sous-réseau 4 : 192.168.10.192/26  hôtes .193 → .254 broadcast .255

On soustrait 2 : l'adresse réseau (bits hôte tous à 0) et l'adresse broadcast (bits hôte tous à 1). Ces deux adresses ne peuvent pas être assignées à un équipement.

09 DNS — Domain Name System

Le DNS est l'annuaire d'Internet. Il traduit les noms de domaine lisibles par l'humain (google.com) en adresses IP compréhensibles par les machines (142.250.74.46). Sans DNS, il faudrait mémoriser les IPs de chaque service.

La résolution est hiérarchique et récursive : le résolveur de votre FAI interroge les serveurs racines (.), puis les serveurs de domaine de premier niveau (.com, .fr…), puis les serveurs autoritaires du domaine cible.

# Interroger le DNS en ligne de commande
$ nslookup google.com                   # résolution simple
$ dig google.com A                       # enregistrement IPv4
$ dig google.com AAAA                    # enregistrement IPv6
$ dig google.com MX                      # serveurs de messagerie
$ dig google.com TXT                     # SPF, DKIM, DMARC...
$ dig -x 142.250.74.46                   # résolution inverse (PTR)
$ dig @8.8.8.8 google.com               # forcer le serveur DNS Google

# Vider le cache DNS local
$ sudo systemd-resolve --flush-caches   # Linux
$ ipconfig /flushdns                     # Windows

10 DHCP — Attribution automatique des IPs

Le DHCP (Dynamic Host Configuration Protocol) permet à un serveur d'attribuer automatiquement une configuration réseau complète aux machines qui se connectent : adresse IP, masque de sous-réseau, passerelle par défaut, serveurs DNS.

Sans DHCP, chaque équipement devrait être configuré manuellement — impraticable dans une entreprise de plusieurs centaines de postes.

# Le processus DHCP en 4 étapes — acronyme DORA

Discover → Client diffuse en broadcast    # "Y a-t-il un serveur DHCP ?" (port 68→67)
Offer    → Serveur répond en unicast/BC    # "Je t'offre 192.168.1.50, bail 24h"
Request  → Client demande officiellement   # "J'accepte l'offre du serveur X"
Ack      → Serveur confirme               # "C'est à toi jusqu'à 14h demain"

# Renouveler son bail DHCP manuellement
$ sudo dhclient -r eth0     # Linux : libérer le bail
$ sudo dhclient eth0         # Linux : demander un nouveau bail
$ ipconfig /release          # Windows : libérer
$ ipconfig /renew            # Windows : renouveler
// bail DHCP et réservations
Une adresse DHCP est temporaire — c'est un bail (lease). Pour les équipements qui doivent toujours avoir la même IP (serveurs, imprimantes, cameras IP), on utilise des réservations DHCP : l'adresse IP est fixée en fonction de l'adresse MAC de l'équipement. C'est plus souple qu'une IP statique car géré centralement.
// DHCP Snooping
Un serveur DHCP malveillant sur votre réseau peut distribuer de fausses configurations et intercepter le trafic (attaque MITM). La fonction DHCP Snooping des switches managés filtre les réponses DHCP : seuls les ports "de confiance" (trusted) peuvent envoyer des offres DHCP.
