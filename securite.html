Sécurité Réseau

Connaître les attaques pour mieux défendre. Chaque vulnérabilité a une contre-mesure. Apprenez les deux.

01 Principales attaques réseau

Les attaques exploitent les failles des protocoles, des configurations, des logiciels ou du facteur humain. Voici les menaces les plus courantes et les plus dangereuses :

Phishing / Spear Phishing

E-mails ou sites frauduleux imitant des entités légitimes pour voler identifiants, mots de passe ou données bancaires. Le spear phishing cible une victime spécifique avec des infos personnalisées.

Fréquence
95%

Ransomware

Logiciel malveillant qui chiffre toutes les données et réclame une rançon (souvent en cryptomonnaie) pour les restituer. Les sauvegardes hors-ligne sont la seule vraie protection.

Impact
95%

DDoS

Saturation d'un serveur ou d'une infrastructure par des millions de requêtes simultanées via un botnet. Rend le service indisponible sans même avoir besoin de l'infiltrer.

Danger
85%

Man-in-the-Middle (MITM)

L'attaquant se positionne entre deux parties qui communiquent, intercepte et peut modifier les échanges à leur insu. Souvent sur Wi-Fi public.

Danger
80%

Brute Force / Credential Stuffing

Tentative systématique de tous les mots de passe possibles, ou utilisation de bases de données de mots de passe volés (credential stuffing). Ciblé sur SSH, RDP, interfaces web.

Danger
75%

SQL Injection

Injection de code SQL malveillant dans un formulaire ou une URL pour accéder, modifier ou supprimer des données en base. Toujours dans le OWASP Top 10.

Danger
80%

ARP Spoofing / Poisoning

Envoi de fausses réponses ARP pour associer l'adresse MAC de l'attaquant à l'IP de la passerelle. Tout le trafic passe alors par l'attaquant (MITM local).

Danger
70%

DNS Spoofing / Cache Poisoning

Corruption du cache DNS d'un résolveur pour rediriger les utilisateurs vers un site malveillant même en tapant la bonne URL. DNSSEC protège contre cela.

Danger
72%

Rogue DHCP / AP

Serveur DHCP ou point d'accès Wi-Fi malveillant sur le réseau local qui distribue de fausses configurations (fausse passerelle, faux DNS) pour intercepter le trafic.

Danger
65%

Zero-Day

Exploitation d'une vulnérabilité inconnue du fabricant ou non encore corrigée. Aucun patch disponible au moment de l'attaque — détection uniquement comportementale.

Impact
90%
// rappel légal
Ces informations sont présentées à des fins éducatives et défensives uniquement. Toute tentative d'intrusion sur un système sans autorisation explicite est un délit pénal passible d'amendes et d'emprisonnement (loi Godfrain en France, CFAA aux USA).

02 Anatomie d'une attaque Man-in-the-Middle

L'attaque MITM est particulièrement insidieuse car les deux victimes ne réalisent pas qu'une tierce partie intercepte leurs communications. Voici comment elle se déroule en pratique :

# Scénario : attaque ARP Spoofing sur réseau local

# Situation normale
Victime (192.168.1.10)  ←→  Routeur (192.168.1.1)

# Après ARP Spoofing par l'attaquant (192.168.1.99)
Victime ARP cache : 192.168.1.1 → MAC attaquant
Routeur ARP cache : 192.168.1.10 → MAC attaquant

# Résultat : tout le trafic passe par l'attaquant
Victime → Attaquant → Routeur → Internet
Victime ← Attaquant ← Routeur ← Internet

# Protections :
# - HTTPS (TLS) rend l'interception illisible
# - VPN chiffre tout le trafic
# - ARP inspection sur les switches managés
# - HSTS strict sur les serveurs web

03 Protections fondamentales

01

Segmentation avec des VLANs

Isoler les équipements par zones fonctionnelles (bureaux, serveurs, invités, IoT, VoIP). Si un équipement est compromis, l'attaquant ne peut pas se déplacer latéralement vers d'autres segments sans passer le firewall.

02

Firewall avec politique "deny all by default"

Tout ce qui n'est pas explicitement autorisé est refusé. C'est le principe du moindre privilège appliqué au réseau. Définir des règles précises par source, destination, port et protocole.

03

Chiffrement des communications

HTTPS / TLS 1.3 pour le web. WPA3 pour le Wi-Fi. VPN pour les accès distants. SSH pour l'administration. Ne jamais transmettre de données sensibles en clair sur le réseau.

04

Authentification forte (MFA)

Activer l'authentification multi-facteurs sur tous les comptes critiques. Un mot de passe volé seul ne suffit plus pour se connecter. Utiliser RADIUS + 802.1X pour l'accès réseau en entreprise.

05

Mise à jour systématique

90% des attaques exploitent des vulnérabilités connues avec un patch disponible. Appliquer les mises à jour critiques dans les 24–72h. Automatiser le patching des systèmes non critiques.

06

Monitoring et alertes

Centraliser les logs (SIEM), surveiller les anomalies de trafic, configurer des alertes sur les événements suspects : connexions en dehors des heures, tentatives de brute force, communication vers des IPs malveillantes.

04 VPN — Virtual Private Network

Un VPN crée un tunnel chiffré entre deux points, protégeant le trafic des interceptions même sur un réseau non sécurisé (Wi-Fi public, Internet). Il masque également l'adresse IP réelle de l'utilisateur.

WireGuard est devenu le standard de facto pour les nouveaux déploiements VPN. Il est intégré dans le noyau Linux depuis 5.6, supporte Windows, macOS, iOS et Android, et offre des performances significativement supérieures à OpenVPN.

05 IDS / IPS — Détection et Prévention d'Intrusions

Un IDS (Intrusion Detection System) analyse le trafic réseau et génère des alertes lorsqu'il détecte des comportements suspects. Il ne bloque rien — il observe et alerte.

Un IPS (Intrusion Prevention System) va plus loin : il analyse le trafic en temps réel et peut bloquer automatiquement le trafic malveillant avant qu'il n'atteigne sa cible.

# Exemple de règle Snort/Suricata (IDS/IPS open source)

# Détecter un scan de ports (nmap)
alert tcp any any -> $HOME_NET any (
    msg:"SCAN nmap SYN scan";
    flags:S;
    threshold: type both, track by_src, count 20, seconds 1;
    sid:1000001;
)

# Bloquer les tentatives de connexion SSH en brute force
drop tcp any any -> $HOME_NET 22 (
    msg:"SSH brute force attempt";
    threshold: type both, track by_src, count 5, seconds 60;
    sid:1000002;
)

07 Bonnes pratiques quotidiennes

✓

Mots de passe forts + gestionnaire de mots de passe

Minimum 16 caractères aléatoires, unique par service. Utilisez un gestionnaire (Bitwarden, KeePass, 1Password) — il est impossible de mémoriser des dizaines de mots de passe forts différents.

✓

Authentification à deux facteurs partout

Activez le MFA sur tous les comptes critiques (email, banque, VPN, SSH). Préférez une application TOTP (Authy, Google Authenticator) à un SMS susceptible d'être intercepté (SIM swapping).

✓

Sauvegardes régulières avec la règle 3-2-1

3 copies des données, sur 2 supports différents, dont 1 copie hors-site (ou hors-ligne pour contrer les ransomwares). Testez la restauration régulièrement — une sauvegarde non testée ne vaut rien.

✓

Principe du moindre privilège

Chaque utilisateur, service ou application ne doit avoir que les droits strictement nécessaires à sa fonction. Un compte compromis avec des droits limités cause beaucoup moins de dégâts.

✓

Former les utilisateurs au phishing

80% des incidents de sécurité commencent par un e-mail de phishing. Des simulations régulières (PhishMe, GoPhish) sont plus efficaces que n'importe quelle formation théorique.

✓

Plan de réponse aux incidents (IR Plan)

Définir à l'avance : qui fait quoi, comment isoler un segment compromis, qui contacter (juridique, assurance, ANSSI en France). Ne pas improviser sous pression — chaque minute compte lors d'un incident majeur.

// philosophie fondamentale
La sécurité n'est pas un état qu'on atteint — c'est un processus continu. Un réseau "sécurisé" aujourd'hui peut être vulnérable demain si les systèmes ne sont pas patchés, les configurations auditées et les équipes formées. La sécurité parfaite n'existe pas : l'objectif est de rendre l'attaque suffisamment coûteuse pour que l'attaquant passe à une cible plus facile.
